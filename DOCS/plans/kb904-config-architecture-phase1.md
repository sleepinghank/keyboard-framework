## 3. 当前设计问题识别

### 3.1 配置项耦合问题

**问题 1: 配置项分散在同一文件中**
- `product_config.h` 混合了：硬件配置 + 行为配置 + 功能开关
- 影响：修改任何一项都需要理解整个文件

**问题 2: 硬件配置与业务配置未分离**
- GPIO 引脚定义与超时时间在同一文件
- 影响：更换硬件平台需要修改包含业务逻辑的文件

**问题 3: 产品区分使用条件编译**
- `#if PRODUCT_ID == 0x0904` 分散在多个文件
- 影响位置：indicator_config.h, output_service.c
- 问题：新增产品需要修改多处

**问题 4: 缺乏配置继承机制**
- 相似产品无法复用配置
- 每个新产品需要完整配置文件

### 3.2 配置项命名不一致

| 当前命名 | 问题 |
|----------|------|
| `CHIP_TYPE_CH584` vs `CHIP_TYPE` | 命名风格不统一 |
| `SDA_PIN` vs `TOUCH_SDA` | 前缀不统一 |
| `BLE_NAME` vs `PRODUCT` | 语义不清晰 |

## 4. HAL 层配置抽象评估

### 4.1 现有 HAL 结构

```
hal/
├── gpio.h              # GPIO 接口定义
├── pin_defs.h          # 引脚定义（支持平台覆盖）
├── platforms/
│   ├── ch584/
│   │   └── _pin_defs.h  # CH584 引脚定义
│   ├── nrf52/
│   │   └── gpio_hal.h   # NRF52 GPIO 实现
│   ├── pixart2860/
│   │   └── gpio_hal.h   # PixArt GPIO 实现
│   └── test/
│       └── _pin_defs.h  # 测试平台引脚
```

### 4.2 HAL 层优点

- GPIO 操作通过统一接口（setPinInput, writePinHigh 等）
- 引脚编号通过 `pin_t` 类型抽象
- 支持平台特定的 `_pin_defs.h` 覆盖

### 4.3 HAL 层不足

- 外设配置（PWM通道、I2C通道）未统一抽象
- 缺乏配置验证机制
- 没有配置冲突检测

## 7. 阶段一结论

### 7.1 主要问题

1. **配置分散**：单个文件混合多种配置类型
2. **耦合度高**：硬件配置与业务逻辑混合
3. **扩展性差**：新增产品需修改多处
4. **缺乏验证**：无配置完整性检查

### 7.2 优化方向

1. 按产品集中组织配置文件
2. 分离硬件配置与业务配置
3. 建立配置继承/默认值机制
4. 添加编译时配置验证
