# CH584 蓝牙全功能联调设计（广播/连接/回连/断联）

日期：2026-03-05

## 1. 目标与边界

### 1.1 目标
- 在 CH584 平台跑通蓝牙完整链路：广播、连接、回连、断联、断联后重新广播。
- 连接后必须验证 HID 键盘上报可用。
- 上电和断联后进入统一恢复流程：回连+广播窗口（60s），超时未连上进入深度休眠，按键唤醒后再次执行恢复流程。

### 1.2 约束
- 遵循现有架构：`Application -> Middleware -> Drivers -> HAL`。
- 最小改动原则，避免修改无关模块。
- 蓝牙驱动仅提供能力与事件，不承载业务策略。
- 联调阶段可加临时日志；任务完成后必须清理。
- 最近连接主机需要掉电记忆。

## 2. 架构设计

### 2.1 职责划分
- HAL：只保留系统基础初始化，不直接做 BLE 协议栈/HID 应用初始化。
- Drivers(Bluetooth CH584)：提供广播、回连、断联、事件上报能力。
- Middleware(wireless)：作为业务策略层，决定何时回连、广播时长、何时休眠。
- Application Service：保留现有任务调度与回调组织，不下沉到驱动细节。

### 2.2 初始化迁移
- 从 `hal/platforms/ch584/_system_hal.c` 移除 BLE 专项初始化调用。
- 在 `system_init_drivers()` 阶段触发 `bt_driver_init(false)`。
- `bt_driver_init()` 增加一次性保护，防止重复初始化。

## 3. 状态与事件（优先复用现有定义）

### 3.1 状态
优先复用 `wt_state_t`：
- `WT_RESET`
- `WT_INITIALIZED`
- `WT_PARING`
- `WT_RECONNECTING`
- `WT_CONNECTED`
- `WT_DISCONNECTED`
- `WT_SLEEP`（若当前无该状态，最小增补）

### 3.2 事件
优先复用现有无线事件定义（`wireless_event_type.h` + 现有 `WL_*` 任务事件）。
仅当现有语义不足时才最小增补事件。

## 4. 业务流程（由 Middleware 决策）

1. 上电/按键唤醒/断联后：进入恢复流程（`WT_RECONNECTING`）。
2. 从持久化读取最近主机索引。
3. 下发驱动动作：发起回连，并开启广播窗口（60s）。
4. 若连接成功：迁移到 `WT_CONNECTED`，允许 HID 上报，并更新最近主机持久化。
5. 若 60s 超时未连接：调用 `system_hal_enter_sleep(SYSTEM_POWER_MODE_DEEP_SLEEP, SYSTEM_WAKEUP_GPIO)`，进入休眠态。
6. 下次按键唤醒后重复上述流程。

## 5. 错误处理与竞态

- 重复初始化：驱动返回并记录错误，不二次拉起协议栈。
- 非法状态迁移：忽略并记录，避免状态机抖动。
- 超时与连接竞态：超时触发前先检查是否已 `WT_CONNECTED`。
- 休眠失败回退：回到 `WT_DISCONNECTED` 并重新进入恢复流程。
- 持久化异常：最近主机读取失败时回退默认槽位，不阻塞广播。

## 6. 验证计划（分阶段编译烧录）

### 阶段A：初始化迁移
- 验证点：BLE 启动正常，且无重复初始化。

### 阶段B：广播窗口
- 验证点：可扫描；60s 未连接后进入深度休眠。

### 阶段C：连接+HID
- 验证点：成功连接且按键上报可被主机接收。

### 阶段D：断联恢复
- 验证点：断联后重新进入恢复流程并可再次连接。

### 阶段E：掉电记忆+唤醒重入
- 验证点：重启后按最近主机策略执行；休眠后按键唤醒可再次恢复。

## 7. 实施原则

- 所有业务策略（是否回连、是否广播、时长、是否休眠）在 middleware。
- 蓝牙驱动仅暴露动作与事件。
- 每完成一个阶段即编译+烧录+串口验收。
- 完成全部任务后，删除联调期新增日志代码。
