# 触控板 PTP 简化流控方案设计

**日期：** 2026-04-02
**状态：** 待评审
**范围：** BLE 触控板 PTP 报文发送路径
**相关文件：**
- `component/touch_component/touchpad_service.c`
- `middleware/communication/wireless.c`
- `middleware/communication/report_buffer.c`
- `application/service/communication_service.c`
- `drivers/communication/bluetooth/ch584/_bt_driver.c`

---

## 1. 背景

当前触控板 PTP 报文已从“直接调用 BT 驱动发送”切换为“通过 `wireless_send_ptp()` 入 `report_buffer`，再由 `REPORT_SEND_EVT` 异步发送”。

实测现象：

- 连接建立后，滑动触控板会持续出现 `[BT] send_ptp fail, ret=22,len:19`
- 失败通常出现在连续滑动一段时间后，而不是第一次触摸即失败
- 键盘/普通鼠标路径问题不明显，PTP 连续流量下问题突出

结合 SDK 定义可知：

- `ret=22` 对应 `blePending (0x16)`
- 含义是前一笔链路事务尚未处理完成，当前发送遇到忙态

因此，本问题本质上不是“PTP 报文格式错误”，而是“连续触控流量进入统一缓冲后，缺少针对高频时效型数据的背压控制与窗口管理”。

---

## 2. 根因分析

### 2.1 当前实现的关键问题

1. `wireless_send_ptp()` 每次都直接入通用队列，PTP 被当成普通可靠报文处理。
2. PTP 属于高频连续状态流，旧帧价值会快速衰减，不适合长队列堆积。
3. `report_buffer_task()` 对 PTP 没有独立节流策略，只按统一发送节拍尝试下发。
4. 当 BLE 链路尚有未完成通知时，继续发送会得到 `blePending`。
5. 当前 `REPORT_SEND_EVT` 续调度只看“队列是否非空”，未完整覆盖“当前报文仍需 retry”的场景。
6. `wireless_send_ptp()` 接收了 `len`，但入队后未保存，最终发送时固定使用 `sizeof(report_ptp_t)-1`，长度语义不完整。
7. `report_buffer_enqueue()` 失败时上层无处理，PTP 在高频场景下可能静默丢帧。

### 2.2 为什么 PTP 更早暴露问题

键盘、消费者键、系统键属于低频离散事件：

- 事件数量少
- 单帧语义强
- 允许进入可靠队列并等待发送窗口

PTP 则是连续运动流：

- 帧率高
- 帧间强替代关系明显
- 更容易在窗口控制缺失时把链路忙态放大出来

因此，当前阶段的判断不是“PTP 必须立刻改成独立语义”，而是：

- 先补齐键盘与 PTP 共同缺失的发送窗口控制
- 再根据测试结果决定是否对特定类型单独优化

## 3. KB04122-13A-WCH_code.xml 可借鉴点

参考项目的 BLE HID 路径具备两层状态：

1. **待发送队列**
   - 新报文先入环形缓冲
   - 由 `SEND_PACKET_EVT` 逐个送入 `HidDev_Report()`

2. **已发送但未确认窗口**
   - 发送成功后不立刻逻辑删除
   - 通过 `LL_GetNumberOfUnAckPacket()` 轮询链路未确认包数
   - 仅当未确认包数下降后才真正回收窗口

这套设计的价值在于：

- 它明确区分了“调用发送接口成功”与“链路已腾出窗口”
- 它把 BLE 忙态视为正常流控信号，而不是异常日志

### 3.1 本工程适合借鉴的部分

- 借鉴“发送窗口”概念
- 借鉴 `LL_GetNumberOfUnAckPacket()` 作为链路背压观测手段
- 借鉴“高频业务不应盲目继续压入链路”的思路

### 3.2 本工程不建议照搬的部分

- 不照搬其 `CMD_* + payload` 的旧封装格式
- 不照搬其完整 HID 混合队列实现

原因：

- 当前工程已采用分层后的 `wireless_transport` 路径
- 直接搬运复杂队列会增加改动面和维护成本
- 当前阶段希望保持键盘与触控板统一发送语义，先验证共性窗口控制是否足够

---

## 4. 设计目标

本次简化流控方案目标如下：

1. 消除连续滑动时大量 `blePending` 失败日志
2. 保证触控板在 BLE 下滑动连续、延迟可控
3. 尽量复用当前 `wireless` / `report_buffer` 架构，减少改动面
4. 为后续需要时演进到完整发送窗口模型预留接口

非目标：

- 本次不重写整套 report_buffer
- 本次不引入 KB04122 式完整通用未确认队列
- 本次不改 HID 描述符与 PTP 数据格式

---

## 5. 简化流控总方案

### 5.1 当前阶段的明确策略

本次先按以下原则调整：

1. **不主动删除旧帧**
   - 当前阶段不采用 latest-slot 覆盖旧帧
   - 不以“保留最新一帧”作为主流控策略

2. **键盘与触控板统一语义**
   - 键盘与 PTP 都继续走同一套 `report_buffer + REPORT_SEND_EVT`
   - 差异先体现在报文长度和上层产生命率，不在发送语义上拆分

3. **统一补发送窗口控制**
   - 重点不是改成“只保留最新帧”
   - 而是给当前通用队列补上“已发送但未确认窗口”的管理

4. **统一保留最新帧镜像**
   - 键盘与触控板都额外保留各自最新帧镜像
   - 该镜像本阶段只用于观测、调试和后续策略切换预留
   - 不用于当前发送路径覆盖旧帧

5. **后续再决定是否退化为 latest-only**
   - 如果统一队列 + 窗口控制后仍无法满足 PTP 连续性
   - 再评估是否对特定类型启用“只保留最新帧”

### 5.2 为什么这是当前工程最合适的方案

相比直接把 PTP 改成 latest-slot，本方案更适合当前阶段：

- 与用户当前要求一致，不主动丢帧
- 键盘与触控板保持统一发送语义，更容易测试和比对
- 直接借鉴 KB04122 的核心价值，也就是“未确认窗口管理”
- 改动集中在 `report_buffer` / `wireless` / `communication_service`，不需要新开一条专用发送通道
- 如果后续测试证明 PTP 必须 latest-only，再切换时也已有最新帧镜像可用

## 6. 方案细节

### 6.1 通用发送状态增强

建议在 `report_buffer.c` 附近新增一组**通用发送状态**，而不是只给 PTP 单独设计：

- `current_report_valid`
  - 当前是否存在“已出队但尚未完成发送生命周期”的报文
- `current_report`
  - 当前正在处理的报文内容
- `current_report_len`
  - 当前报文真实发送长度
- `current_inflight`
  - 当前报文是否已成功送入 BLE，但链路窗口尚未释放
- `current_last_send_time`
  - 当前报文最近一次提交发送的时间

这样做的意义：

- 键盘与触控板共用一套语义
- 把“出队成功”和“真正可以处理下一帧”明确区分开

### 6.2 通用队列继续保留，不覆盖旧帧

当前 `wireless_send_keyboard()`、`wireless_send_ptp()` 的基础方向保持不变：

- 仍然构造 `report_buffer_t`
- 仍然进入 `report_buffer_queue`
- 仍然由 `REPORT_SEND_EVT` 统一驱动发送

但需要补两点：

1. 队列项中必须保存真实 `payload_len`
   - 尤其是 PTP，不能入队后再固定写死为 `sizeof(report_ptp_t)-1`

2. 入队失败不能静默
   - 队列满时不覆盖旧帧
   - 保持当前队列内容不变
   - 记录计数或日志，便于后续判断是否需要退化为 latest-only

### 6.3 统一保留最新帧镜像

虽然本阶段不删除旧帧，但可以为键盘与触控板都保留一份最新帧镜像：

- `latest_keyboard_report` / `latest_keyboard_len`
- `latest_ptp_report` / `latest_ptp_len`

或者做成统一结构，按 `REPORT_TYPE_*` 维护最新镜像。

本阶段这份镜像的用途是：

- 排查“当前队列尾部是否已经落后于实时输入”
- 评估后续是否需要把某一类报文改成 latest-only
- 在异常恢复时辅助判断是否需要补发最新状态

**注意：**

当前阶段这份最新镜像不替代队列，不参与覆盖旧帧。

### 6.4 `report_buffer_task()` 改为窗口驱动

建议把 `report_buffer_task()` 改成如下顺序：

1. 如果 `current_inflight == true`
   - 先检查链路窗口是否已释放
   - 若未释放，则本轮不再出队新报文

2. 如果 `current_report_valid == false`
   - 再从队列中取下一条报文放入 `current_report`

3. 对 `current_report` 尝试发送
   - 根据类型调用 `send_keyboard` / `send_mouse` / `send_ptp` 等

4. 按返回状态更新当前报文状态
   - 成功：进入 `current_inflight`
   - 临时忙态：保留 `current_report_valid`，下轮重试
   - 致命失败：记录并按现有策略处理

这套机制的核心是：

- 队列中的下一帧只有在“当前帧不再占用链路窗口”时才允许发送

### 6.5 `blePending` 的处理语义

当发送返回以下状态时：

- `blePending`
- `bleNoResources`
- `bleTimeout`

统一视为**临时忙态**：

- 不当成真正逻辑错误
- 不清除 `current_report`
- 不继续出队下一条
- 只在下一个发送窗口再试当前报文

这样处理后：

- 键盘和触控板的忙态语义一致
- 不会因为 PTP 连续上报而越忙越发

### 6.6 链路窗口释放策略

本次方案同样给出两个级别，但现在是**通用队列共用**，不只针对 PTP：

#### 方案 A：时间窗版

- 成功调用发送接口后
- 按固定时间窗延后下一次发送
- 在时间窗到达前，不再处理下一条队列报文

优点：

- 实现最小

缺点：

- 仍然是估算，不是真实链路窗口

#### 方案 B：未确认包数版

- 成功发送后，读取 `LL_GetNumberOfUnAckPacket(hidEmuConnHandle)`
- 若未确认包数仍大于 0，则保持 `current_inflight = true`
- 仅当未确认包数归零后，才允许处理下一条队列报文

优点：

- 最接近 KB04122 的真实流控
- 对键盘与触控板都有效

缺点：

- 需要补一层链路查询封装

本设计建议优先落地 **方案 B**。

### 6.7 `REPORT_SEND_EVT` 续调度条件补齐

当前 `REPORT_SEND_EVT` 续调度只看队列是否非空，这还不够。

建议改成以下任一条件满足就继续调度：

- `!report_buffer_is_empty()`
- `report_buffer_get_retry() > 0`
- `current_report_valid == true`
- `current_inflight == true`

这样可以确保：

- 当前报文临时忙态时不会丢失后续重试机会
- 已发送但窗口未释放时，系统仍能继续轮询窗口状态

### 6.8 忙态日志处理

`bt_driver_send_ptp()`、`bt_driver_send_keyboard()` 等发送接口对以下状态应降低日志噪音：

- `blePending`
- `bleNoResources`
- `bleTimeout`

建议：

- 忙态只做低频日志或计数统计
- 真正异常才打印错误

否则连续输入时日志会放大问题表象，影响定位。

## 7. 与当前 report_buffer 的边界划分

### 7.1 保持走统一通用队列的类型

- 键盘
- 消费者键
- 系统键
- 鼠标
- 触控板 PTP

本阶段不单独拆 PTP 通道。

### 7.2 额外维护最新帧镜像的类型

- 键盘
- 触控板 PTP

理由：

- 这两类最需要后续对比“实时状态”和“队列尾部状态”
- 后续若要切换策略，能最小化改动

## 8. 实施步骤建议

### 阶段 1：统一窗口控制闭环

1. 为 `report_buffer_t` 增加真实发送长度字段
2. 为 `report_buffer_task()` 增加 `current_report_valid/current_inflight` 通用状态
3. `wireless_send_keyboard()`、`wireless_send_ptp()` 继续入队，但同步更新各自最新帧镜像
4. `REPORT_SEND_EVT` 续调度条件补齐 `retry/current_report/inflight`
5. `bt_driver_send_*()` 对忙态日志降噪

预期结果：

- 键盘与触控板共用一套更稳定的发送节奏
- 连续滑动时 `ret=22` 日志显著减少
- 当前阶段不通过删除旧帧来换取稳定性

### 阶段 2：真实窗口增强

1. 增加 `LL_GetNumberOfUnAckPacket()` 查询封装
2. 把固定时间窗升级为真实未确认包窗口判断
3. 让下一条队列报文严格等待窗口释放后再发送

预期结果：

- 行为更接近 KB04122 参考实现
- 队列节奏更贴合真实 BLE 链路能力

### 阶段 3：策略评估

在阶段 1 和阶段 2 完成后，基于测试结果判断：

- 队列是否仍会在 PTP 高频滑动下持续堆积
- 键盘输入是否受影响
- 最新帧镜像与实际发送尾帧是否存在持续偏差

只有当这些测试表明确有必要时，才进入：

- 针对特定类型启用 latest-only
- 或仅对 PTP 启用覆盖式发送

## 9. 验证建议

### 9.1 功能验证

1. 单指连续滑动 30 秒
   - 预期：光标/手势连续，无明显卡顿
2. 两指滑动/滚动
   - 预期：报文持续正常，无明显跳变
3. 触摸后快速抬起
   - 预期：末帧释放可及时送达
4. 长时间连接后再滑动
   - 预期：不出现首帧长时间堵塞

### 9.2 日志验证

1. 连续滑动时不应持续刷 `[BT] send_ptp fail, ret=22`
2. 若存在忙态日志，应为低频可观测，而非每帧打印

### 9.3 压力验证

1. 持续快速滑动
2. 滑动同时穿插键盘输入
3. 滑动同时发生 PMU 状态切换

目标是确认：

- PTP 不压垮键盘输入
- 键盘输入也不会使 PTP 阻塞失控

## 10. 风险与取舍

### 风险 1：当前阶段不删帧，队列仍可能在极端输入下增长

这是当前阶段主动接受的取舍。

原因：

- 当前目标是先验证“统一窗口控制”能否解决主问题
- 在没有充分测试前，不希望通过丢帧改变输入语义

### 风险 2：固定时间窗与真实连接窗口不完全一致

因此建议第二阶段接入 `LL_GetNumberOfUnAckPacket()`，从时间驱动升级为真实窗口驱动。

### 风险 3：队列满时仍然可能出现入队失败

本阶段不以覆盖旧帧解决该问题，因此必须：

- 补日志或计数
- 在测试中重点观察是否真的达到队列上限

### 风险 4：键盘与触控板统一语义后，某一类高频报文可能拖慢另一类

这也是本阶段要重点测试的内容。

如果后续证明确实存在此问题，再基于最新帧镜像决定是否对某一类报文单独退化。

---

## 11. 结论

当前工程最合适的简化流控方案不是直接改成“只保留最新帧”，而是：

**先借鉴 KB04122 的“链路窗口管理”思想，在当前通用 `report_buffer` 上补齐统一的 inflight 控制、忙态重试与续调度条件；键盘和触控板都不主动删帧，只额外保留最新帧镜像作为后续策略切换预留。**

这套方案具备以下优点：

- 满足当前“先不删帧”的要求
- 键盘与触控板保持统一语义，便于验证
- 能先验证真正根因是否就是窗口控制缺失
- 若后续仍有问题，可平滑演进到 latest-only，而不必重做接口

建议评审结论：

- **优先实施：** 通用 inflight 窗口控制 + 续调度补齐 + 忙态降噪 + 最新帧镜像
- **推荐增强：** 引入 `LL_GetNumberOfUnAckPacket()` 做真实窗口判定
- **后续备选：** 若测试仍不理想，再评估对特定类型启用“只保留最新帧”
