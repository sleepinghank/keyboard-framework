# Phase 1 Workflow — 需求确认（步骤 1-3）

> 本文件是 design-first skill 的阶段一详细流程。

## 本阶段目标

确认需求、理解现状，产出结构化交接文档供阶段二使用。

**上下文管理规则：**
- 本阶段只在对话中保留"当前步骤所需信息"
- 步骤完成后立即将结论写入文件，从对话中卸载
- **禁止**在后续步骤中重复展开已完成步骤的详细内容

---

## 步骤 1 — 探索项目上下文

声明：`【步骤 1/9 — 探索项目上下文】`

**执行内容：**
- 检查文件结构、文档、近期提交记录
- 识别与需求相关的现有模块（仅标记路径，不深入阅读）
- 评估请求规模：
  - 单一功能 → 正常推进
  - 多个独立子系统 → 立即标记，协助用户拆解为子项目

**输出格式（精简）：**
```
项目根目录: <path>
相关模块: [模块A: path, 模块B: path, ...]
规模判断: 小/中/大
```

> **[CHECKPOINT 1 — 必须暂停]**
>
> 将探索结果以上述格式展示给用户。
>
> 明确说："步骤 1 完成。是否继续步骤 2（需求澄清）？"
>
> **在此停止，等待用户明确回复后才能继续。**

---

## 步骤 2 — 需求澄清与 Agent 评审

声明：`【步骤 2/9 — 需求澄清与 Agent 评审】`

### Part A：逐一提问

- 每次只问一个问题，提问后停止等待回答
- 能给选项时优先给选项
- 聚焦三点：目的、约束条件、成功标准
- 所有问题回答完毕后才能进入 Part B

### Part B：Agent 需求评审团（并行输出）

所有问题回答完毕后，以下三个角色**同时**输出评审意见：

| Agent 角色 | 关注点 | 输出格式 |
|-----------|--------|---------|
| 产品视角 | 功能是否满足使用场景？有无遗漏的用户期望？ | 风险点列表（≤3条） |
| 架构视角 | 技术可行性、与现有架构兼容性、潜在技术风险 | 风险点列表（≤3条） |
| 质量视角 | 如何验证完整性？边界条件和异常路径 | 验证点列表（≤3条） |

**评审输出示例：**
```
产品视角：[风险1], [风险2]
架构视角：[风险1], [风险2]
质量视角：[验证点1], [验证点2]
```

### Part C：写入文件

将以下内容写入 `docs/plans/<topic>-requirements.md`：
- 已确认需求点（含用户回答）
- Agent 评审意见汇总
- 范围边界（在范围内 / 不在范围内）

> **[CHECKPOINT 2 — 必须暂停]**
>
> 展示文件内容摘要，聚焦于：需求点、风险点、范围边界。
>
> 明确说："步骤 2 完成，需求文件已保存至 `<文件路径>`。是否继续步骤 3（代码现状分析）？"
>
> **在此停止，等待用户明确回复后才能继续。**

---

## 步骤 3 — 代码现状分析（并行 Subagent）

声明：`【步骤 3/9 — 代码现状分析（并行分析中...）】`

### 并行执行策略

根据步骤 1 识别的相关模块，将分析任务拆分给多个 Subagent 并行执行：

**Subagent 分配规则：**
- 每个 Subagent 负责一个独立模块/子系统
- 模块数 ≤ 3：每模块一个 Subagent
- 模块数 > 3：按功能层次合并，保持 Subagent 数量 ≤ 5

**每个 Subagent 的任务：**
```
输入：模块路径 + 需求文件路径
执行：
  1. 读取模块接口定义
  2. 识别与需求相关的关键函数/类
  3. 梳理该模块的数据流
输出（仅结论）：
  - 模块名称
  - 当前能力：[已有功能列表]
  - 缺失能力：[需要新增/修改的列表]
  - 关键接口：[接口签名，≤5个]
  - 依赖关系：[上下游模块]
```

**主线程汇总规则：**
- 收集所有 Subagent 的结论输出
- 不保留 Subagent 的分析过程，只保留结论
- 生成跨模块的数据流图（文本形式）

### 汇总输出格式

```
## 现状分析结论（金字塔原理：结论先行）

### 核心结论
当前缺少：[X, Y, Z]
需要修改：[模块A的接口B, 模块C的函数D]

### 各模块详情
| 模块 | 当前能力 | 缺失能力 | 关键接口 |
|------|---------|---------|---------|
| ... | ... | ... | ... |

### 数据流
[模块A] → [模块B] → [模块C]
```

### 写入文件

将汇总输出追加至 `docs/plans/<topic>-requirements.md`

> **[CHECKPOINT 3 — 阶段一完成，必须暂停]**
>
> 展示现状分析汇总（仅结论部分）。
>
> 明确说："阶段一（需求确认）完成，分析结果已追加至需求文档。是否进入阶段二（方案设计）？"
>
> **在此停止，等待用户明确确认后才能进入阶段二。**
