● 【步骤 3/9 — 代码现状分析】

  结论先行：
  - 当前代码缺少"单按 S_FN_KEY 发 Earth"逻辑
  - 需要修改 kb_combo_map.c（改造 FN_DOWN/UP combo 为多事件）、kb_fn_action.c 或
   kb_sys_action.c（实现 Earth 回调）
  - report_update_proc 无需修改

  ---
  现在逐层分析支撑依据。

  1. 数据流与调用链

  matrix_scan()
      → update_key_code_list()     // 更新 _key_code_list
      → combo_task()               // 遍历所有 combo，运行状态机，触发回调
          → apply_combo()          // 检测组合键是否激活
          → button_ticks()         // 状态机推进（PRESS_DOWN/UP/LONG_PRESS_*）
          → cb[event]()            // 触发回调，结果写入 _key_code_list_extend
      → report_update_proc()       // 合并两个链表，组装 HID 报告并发送

  2. FN_DOWN/UP combo 当前结构

  kb_combo_map.c 中定义：

  // 两个独立 combo，分别监听 PRESS_DOWN 和 PRESS_UP
  [FN_DOWN_ID] = COMBO(FN_DOWN_combo, PRESS_DOWN, FN_DOWN_KEY),
  [FN_UP_ID]   = COMBO(FN_UP_combo,   PRESS_UP,   FN_UP_KEY),

  两个 combo 共用同一个 FN_DOWN_combo[]（{ S_FN_KEY, COMBO_END 
  }），但分别注册不同事件。按下时只触发 FN_DOWN_KEY（设标志位，不发
  HID）。释放时只触发 FN_UP_KEY（清标志位，不发 HID）。

  3. 旧版注释代码分析

  kb_sys_action.c 第 147-186 行有完整的注释实现：

  // Earth_PRESS_UP_Handler：检查 combinations_flag==1，按平台发对应报告
  // Earth_LONG_PRESS_START_UP_Handler：Apple 平台发 M_EARTH（长按开始）

  旧代码缺陷：
  - 未处理 Android 平台（逻辑坑：else 分支用了 KB_L_GUI + KB_SPACEBAR，Android
  应为 Shift+Space）
  - 长按只在 LONG_PRESS_START（开始时）发一次，未处理
  LONG_PRESS_HOLD（持续维持）
  - 场景 C（共用键 + 普通键，Apple）未实现

  4. combo_t 多事件支持能力

  kb_combo_engine.h 中的宏：

  COMBO2(ck, e1,cb1, e2,cb2)                   // 2 个事件
  COMBO3(ck, e1,cb1, e2,cb2, e3,cb3)           // 3 个事件
  COMBO3_LONG_TICKS(ck,ticks, e1,cb1, ...)     // 3 个事件 + 自定义长按阈值

  combo_t.cb 数组大小为 number_of_event（=
  7），理论上支持所有事件都注册回调，但宏定义最多 3 个。本次需要 4
  个事件（PRESS_DOWN / PRESS_UP / LONG_PRESS_START / LONG_PRESS_HOLD），需要新增
   COMBO4_LONG_TICKS 宏，或拆分为两个 combo entry（目前已是两个，继续沿用）。

  5. combinations_flag 标志位语义（关键）

  // kb_sys_action.c
  uint8_t combinations_flag = 1;   // 初始为1

  // kb_fn_action.c
  FN_DOWN_KEY: combinations_flag = 1
  FN_UP_KEY:   combinations_flag = 0

  // kb_combo_engine.c
  active_fn_combo == 1 时: combinations_flag = 0

  combinations_flag == 1 意味着：S_FN_KEY 已按下，且尚未命中 Fn 
  功能组合。这正是判断"是否应该发 Earth"的条件。

  6. 场景 C 在 combo 层的实现路径

  combo_task() 每次 KEY_UPDATE 都会被调用。当 S_FN_KEY 按住（FN_st==1）时，若
  _key_code_list 中出现非 S_FN_KEY 的普通键，且 active_fn_combo==0，则应在 Apple
   平台补发 Earth。

  这个检测需要在 FN_DOWN combo 的 LONG_PRESS_HOLD 回调中（持续被调用）判断
  _key_code_list 内容，或在 combo_task 的主循环中增加一个独立的"Earth 监控
  combo"。

  最简路径：利用 FN_DOWN_combo 的 LONG_PRESS_HOLD
  回调持续检查是否有普通键进入，根据 host_system_type 和 combinations_flag
  决定是否发 Earth。

● 【步骤 4/9 — 差距分析与方案制定】

  核心矛盾

  │                            │ 回调              │                       │
  ├────────────────────────────┼───────────────────┼───────────────────────┤
  │ 场景 C Apple 补发 Earth    │ 无检测机制        │ FN_st                 │
  │                            │                   │ 期间出现普通键时补发  │
  ├────────────────────────────┼───────────────────┼───────────────────────┤

  │                            │ ）                │                       │
  └────────────────────────────┴───────────────────┴───────────────────────┘

  ---
  候选方案

  方案一（推荐）：扩展现有 FN combo，在回调中集中处理 Earth 逻辑

  核心思路： 在 FN_DOWN_combo 上新增 LONG_PRESS_HOLD 回调，在 FN_UP 回调中增加
  Earth/Win+Space 发送逻辑。引入一个模块内状态变量 earth_sent，记录"Earth down
  是否已发出"，供 FN_UP 决定是否需要发 Earth up。

  改动点：
  - kb_combo_map.c：FN_DOWN_ID 改用 COMBO2_LONG_TICKS 同时注册 PRESS_DOWN +
  LONG_PRESS_HOLD；FN_UP_ID 改用 COMBO2 同时注册 PRESS_UP（已有）
  - kb_fn_action.c：FN_UP_KEY 中增加平台判断和 Earth 逻辑；新增 FN_LONG_HOLD
  回调
  - 引入 earth_sent 标志位（在 kb_fn_action.c 模块内部）

  优点： 改动最小，完全在现有 combo 框架内，不新增文件，不修改 combo 引擎
  缺点： FN_UP_KEY 承担了发报告的职责，与"回调只操作标志位"的现有风格略有偏差

  ---
  方案二：新增独立的 Earth 状态机 combo

  核心思路： 保持现有 FN_DOWN_ID/FN_UP_ID 不变，新增一个
  EARTH_COMBO_ID，专门监听 S_FN_KEY
  的完整生命周期（DOWN/UP/LONG_PRESS_HOLD），在回调中处理全部 Earth 逻辑。

  改动点：
  - kb_combo_map.c：新增 EARTH_COMBO_ID entry，使用 COMBO3_LONG_TICKS 绑定 3
  个事件
  - kb_sys_action.c：重新启用并修订注释代码，实现 Earth_DOWN/Earth_UP/Earth_HOLD
   三个回调

  优点： 职责分离清晰，Earth 逻辑与 Fn 标志位逻辑完全隔离
  缺点： 同一个物理键（S_FN_KEY）绑定了两个 combo entry，apply_combo
  会对同一个键计算两次状态，需确认引擎对此无副作用；新增 combo entry 增加了

  核心问题：combo 顺序执行的时序困境

  combo_task 按 combo 数组索引顺序逐个处理。每次处理 combo
  时，都会立即触发回调：

  combo_task 单次执行顺序：
    → active_fn_combo = 0  (重置)
    → [0] FN_DOWN_ID: LONG_PRESS_HOLD 回调触发 ← 此刻根本不知道后面会发生什么
    → [1] FN_UP_ID: ...
    → [2] FN_F1_ID: Bringhness_Down 触发，KC_F1 被消费
    → ...
    → 末尾: active_fn_combo → 0 (F1-F12 是单键
  combo，fn_combo=0，根本不设此标志)
           combinations_flag 根本没有被清零

  还有第二个问题：combinations_flag 对 F1-F12 类 Fn 
  功能键完全失效。active_fn_combo 只对包含 S_FN_KEY 的多键 combo（如
  FN+DEL）生效，F1-F12 是单键 combo，fn_combo=0，所以 combinations_flag 即使
  Fn+F1 触发了也不会被清零。

  结论：现有 combinations_flag 机制无法可靠判断"是否有 Fn 
  功能键被激活"，方案一和方案二都有此问题。

  ---
  修订方案：post-loop Earth 状态机

  核心思路： combo 回调不直接发送 Earth 报告，只维护标志位。在 combo_task
  主循环结束之后，所有 combo 都处理完毕，再做 Earth 决策。

  需要两个新机制：

  ① 检测"Fn 功能键是否激活"

  在 combo_task 主循环中，额外追踪一个 fn_function_fired 标志：

  uint8_t fn_function_fired = 0;
  for (uint8_t i = 0; i < number_of_combos; ++i) {
      ...
      if (combo->event触发 && i != FN_DOWN_ID && i != FN_UP_ID) {
          if (FN_st == 1) fn_function_fired = 1;  // Fn 功能键触发了
      }
  }
  // 此时 fn_function_fired 已准确反映本 tick 内是否有 Fn 功能被使用

  ② post-loop Earth 决策

  主循环结束后，Earth 状态机：

  earth_state = PENDING:
    - fn_function_fired → earth_state = FN_TAKEN（屏蔽 Earth）
    - Apple平台 && _key_code_list 有普通键 → 立即发 Earth down，earth_state =
  EARTH_DOWN
    - （保持 PENDING，等待 FN_UP）

  FN_UP 触发时（FN_st 从1→0）:
    - earth_state == PENDING：发 Earth tap（单按场景）或 Win+Space / Shift+Space
    - earth_state == EARTH_DOWN：发 Earth up（场景C释放）
    - 复位 earth_state = IDLE

  这样：
  - 当 Fn+F1 按下时，fn_function_fired=1 在主循环末尾已被正确设置，post-loop
  看到后设 FN_TAKEN，不发 Earth ✓
  - 当 Fn+普通键（Apple）按下时，fn_function_fired=0，_key_code_list
  有普通键，立即发 Earth ✓
  - 单按释放时，earth_state==PENDING，FN_UP 回调发 Earth tap ✓

  Earth 状态机 + fn_function_fired 检测，替换原来不可靠的 combinations_flag。
